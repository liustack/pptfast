# Roadmap

## v0.4 ŌĆö vocabulary-v4 rename (shipped in 0.4.0)

A pure domain-language rewrite, no behavior change (`.issues/specs/2026-07-19-pptfast-narrative-spec-vocabulary.md`): `scenario` ŌåÆ `narrative`, `mode` ŌåÆ `strategy`, `mode: "narrative"` ŌåÆ `strategy: "storytelling"`, `delivery` ŌåÆ `pacing` (`text`/`presentation` ŌåÆ `dense`/`spacious`, `balanced` unchanged), `plan` ŌåÆ `spec` (`deck.plan.json` ŌåÆ `deck.spec.json`, `pptfast plan validate` ŌåÆ `pptfast spec validate`), page-level `rhythm` ŌåÆ `beat`. `pptfast scenarios` ŌåÆ `pptfast narratives`, `pptfast schema --plan` ŌåÆ `pptfast schema --spec` ŌĆö the old commands fail outright with a pointer to the new one, no long-lived alias. IR v4 is the new default version; `pptfast migrate <input> -o <output>` deterministically converts a v3 IR file or a `deck.plan.json` project directory, field-for-field, no model call. A v4-labeled document that still writes the pre-rename field names or enum values hard-rejects, same as any other unrecognized field or value (spec ┬¦16, reversing an earlier ┬¦15.4 call to alias-rescue it) ŌĆö `pptfast migrate` is the only supported path for old-vocabulary input. See `docs/concepts.md`'s narrative section and README's "Narratives" section for the settled vocabulary, and `docs/deck-projects.md`/`docs/selection-and-seed.md` for where it's consumed.

## v0.3 ŌĆö concept realignment (shipped in 0.3.0)

Realign the vocabulary around a Keynote-style, no-master model and re-home structural selection onto narrative axes. These axes shipped in 0.3.0 named `scenario`/`mode`/`delivery`; the v0.4 vocabulary-v4 rename above renamed them to `narrative`/`strategy`/`pacing` with no behavior change ŌĆö this section describes what shipped in 0.3.0, using the current (v0.4+) names throughout:

- **theme** = the distributable, top-level unit: `style` + `brand` + a layout set for each page type ŌĆö the 13 built-ins become themes, and a style override re-colors a theme. Shipped full-set-by-default (W4): every built-in points each page type at that type's complete registry pool unless curation narrows it, and today none of the thirteen do ŌĆö an earlier three-theme chapter-archetype exclusion (a fixed ink-luminance threshold undershooting 3:1 against one archetype's accent color) was lifted once that threshold was replaced with a real per-background dual-ink contrast comparison, which clears all three comfortably. `registerTheme`'s `layouts` argument is optional for the same reason, defaulting to the full set per page type when omitted
- **theme.style** = pure design tokens (color/type/radius/icon flavor/spacing scale) ŌĆö aligned with both the PPTX theme and the shadcn theme
- **theme.brand** = identity chrome (logo/footer/page number/watermark/cover signature), filled into a layout's optional slots. No slideMaster concept: a layout with no slot for a given brand element simply omits it (graceful degradation) ŌĆö chrome has always been drawn straight into the flat per-slide SVG, so the model now matches the implementation
- **layout** = page-level template with named slots ŌĆö shipped as an explicit registry (30 archetypes + 4 image takeovers), replacing the old archetype/variant split. **component** = the atomic units that fill slots ŌĆö shipped for content (the IR's `components`, renamed from `blocks`). Unifying decoration/media/icon elements under the same concept is still ahead
- **narrative** = strategy ├Ś pacing ├Ś audience ŌĆö shipped in the IR as an optional field: a named preset string (e.g. `boardroom-report`) or a partial axes object, 7 presets each carrying soft theme recommendations, an omitted axis or the whole field defaulting to `general`. `pacing` scales the content-quality gate's density and bullets budgets, and (W4) the body-text baseline: 20/24/32px for dense/balanced/spacious, applied only to the paragraph/bullets/callout trio (every other component's own type scale and the heading system are untouched) ŌĆö `balanced` is the narrative default, so an omitted narrative now renders body text at 24px, not the pre-W4 fixed 20px. `strategy`'s layout tendencies are wired into structural selection too, see the dedicated bullet below ŌĆö a theme's layout set for a page type stays the hard boundary (what this theme supports), narrative only weights inside it, never a hard whitelist
- **Weighted layout selection shipped (W4)**: an omitted slide `layout` resolves through four deterministic steps ŌĆö the page type's full registry pool, the theme's layout set for that page type (full by default, see the theme bullet above), the narrative `strategy`'s soft ├Ś3 weight on a handful of content-layout ids per strategy (`STRATEGY_DEFINITIONS[strategy].layoutTendencies` ŌĆö `TENDENCY_WEIGHT`/`BASE_WEIGHT` in `svg/effective-layout.ts`, cover/chapter/ending never weighted since their character comes from the theme, not the strategy), and a seeded weighted pick (`weightedPickBySeed`, `svg/variety.ts`) with a deterministic swap to the runner-up when it would repeat the immediately preceding slide's layout. A rarer `narrativesOnly` hard filter shipped on the layout registry too (renamed from `scenariosOnly` in the vocabulary-v4 rename's internal-name sweep, spec ┬¦16), for a strategy-exclusive layout ŌĆö unused by any of the 30 built-in archetypes today. Revision stability (editing one page without reshuffling every other page's auto-pick) is a three-tier seed ladder: an explicit `ir.seed` wins outright, a deck project's `assembleDeck` derives and persists one from the spec's filename + page ids when the spec omits `seed` (the CLI surfaces the value to copy in), and a bare IR with neither falls back to a content hash of filename + headings (legacy-compatible, but a heading edit reshuffles every auto-pick deck-wide). `assembleDeck` also materializes every auto-picked layout into its slide's `layout` field in `deck.json` ŌĆö an explicitly pinned page is left untouched
- Deck spec artifact (`deck.spec.json`, named `deck.plan.json` before the v0.4 rename) shipped as a CLI-gated first-class citizen: `pptfast spec validate` enforces strategy-aware hard gates (boundary pages, heading length, beat rotation, page count vs. pacing), and `assembleDeck`/`disassembleDeck` (SDK) plus `assemble`/`disassemble` (CLI) materialize a deck project directory (`deck.spec.json` + `pages/<id>.json` + `assets/`) to and from IR ŌĆö enabling page-level generation and revision without holding a whole deck in context
- Deterministic geometry audit shipped: `pptfast audit <target> [--json]` (SDK: `auditDeck(ir): AuditReport`, pure, zero Node dependency) renders every non-placeholder page off-screen and checks overflow/out-of-bounds (text past its own box or the page), low-contrast (WCAG relative-luminance ratio, text blended through inherited opacity vs. its resolved background region), and overlap (pairwise component bounding-box intersection) ŌĆö advisory, not a hard gate (`validateIr` already is one), agent-judgeable by exit code alone (0 clean, 1 with findings). The same findings now overlay `pptfast preview --html`'s bundle (per-page badges + a findings panel) whenever the deck has no placeholder pages
- Self-contained HTML preview bundle shipped: `pptfast preview <target> -o <dir> --html` (spec ┬¦7 workflow Ōæż) writes a single `preview.html` alongside the existing per-slide SVG files ŌĆö every slide's already-rendered SVG embedded directly into the markup (never `<img src>`, no other external file reference), a bottom thumbnail filmstrip, keyboard (ŌåÉ/ŌåÆ) and click navigation, and a page counter, with placeholder pages carrying a visible "unfilled" badge in both views ŌĆö for a human (or an agent that can view HTML) to flip through a whole deck at once instead of opening N separate SVG files. A pure string builder (`buildPreviewHtml`), CLI-only and not part of the SDK's public surface, assuming every image asset is local or a `data:` URI ŌĆö a remote `http(s)` asset src passes through un-inlined and stays a live network reference in the bundle, a known limitation. The bundle now also carries the read-only review loop: audit-findings overlay (see the audit bullet above) plus per-page free-text annotations exporting a structured `revision-request.json` for the agent to route back through `pages/*.json` edits. Still ahead: `pptfast serve` (a live preview server serving the same loop without the download step)

The v4 IR schema is frozen as of 0.4.0 ŌĆö future evolution is additive only (new optional fields, new enum members), and any breaking change ships under a new top-level `version` value with the same hard-reject-and-migration treatment v3 got.

## Theme ecosystem (v0.5+)

Distribute themes the way shadcn distributes registry items: a theme is a JSON bundle installable from a registry URL via the CLI (`pptfast theme add <url>`). Community registries stay open. Style and curated layouts open first, brand chrome opens later. A curated premium theme marketplace and brand-VI-to-theme services are potential commercial channels ŌĆö the engine itself stays MIT (open-core).
